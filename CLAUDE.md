# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run build          # compile TypeScript to dist/
npm run clean          # remove dist/
npm run start:jamf     # run standalone JAMF MCP HTTP server (port 3001)
npm run start:intune   # run standalone Intune MCP HTTP server (port 3002)
npm test               # compile (tsconfig.test.json) + run integration tests against live JAMF, plus the auth unit tests
npm run test:write     # same, but enables destructive write tests (BlankPush, UpdateInventory, etc.)
npm run test:unit      # auth/roles unit tests only — no live JAMF credentials or network required
```

Start scripts use Bitwarden Secrets Manager — set `BWS_ACCESS_TOKEN` in your shell, then run `./start-jamf.sh` or `./start-intune.sh`. The `bws run --` wrapper injects credentials as env vars.

Tests require credentials as env vars (set by BWS or manually) plus `TEST_COMPUTER_NAME`, `TEST_COMPUTER_SERIAL`, and `TEST_USER_EMAIL` for test fixtures. The package/script/smart-group/policy write tests (see Tests below) additionally read optional `TEST_PACKAGE_PATH`, `TEST_SCRIPT_NAME`, `TEST_SMART_GROUP_NAME`, `TEST_POLICY_NAME` — each test skips gracefully if its var is unset.

## Architecture

Two standalone Streamable HTTP servers built on Express + `@modelcontextprotocol/sdk`. Each is single-file and self-contained — helpers like `toText`/`notFound`/`errorResult`/`resolveDevice` are duplicated per server rather than shared, so a fix in one server file does not propagate to the other:

- **`src/mcp/jamf-server.ts`** — JAMF MCP server on port 3001. `createJamfMcpServer(roles)` builds a fresh `McpServer` + `JamfClient` per HTTP request (stateless mode, required for load-balanced/APIM deployment), registering only the tools the caller's resolved roles permit; every POST to `/mcp` gets its own `StreamableHTTPServerTransport` with `sessionIdGenerator: undefined`, torn down on `res.on("close")`. Exposes read tools plus write tools (`jamf_send_mdm_command`, `jamf_update_computer`, `jamf_assign_computers_to_prestage`, `jamf_flush_mdm_commands`).
- **`src/mcp/intune-server.ts`** — Intune MCP server on port 3002. Same per-request stateless transport pattern. Exposes read tools plus write tools (`intune_send_device_action`, `intune_set_device_category`, `intune_set_device_name`, `intune_set_autopilot_group_tag`, `intune_assign_policy`, `intune_assign_app`, `intune_create_win32_app`, `intune_assign_app_to_groups`).

Both expose `GET /health` for load balancer checks and `GET /metrics` for Prometheus scraping, and return 405 on `GET`/`DELETE /mcp` (stateless servers don't support session resumption). MCP endpoints: `http://localhost:3001/mcp` (JAMF) / `http://localhost:3002/mcp` (Intune)

Both require `Authorization: Bearer <token>` on `/mcp` (not on `/health`), enforced by the shared `src/utils/auth.ts` `requireMcpAuth(options)` middleware — one of three intentionally-shared modules (alongside `src/utils/entra-jwt.ts` and `src/utils/roles.ts`), since duplicating auth/role logic risks the two servers' copies drifting out of sync. `requireMcpAuth` supports EITHER of two independent auth modes on the same request, though only one is enabled in production as of 2026-07-15:
1. A static bearer token from `JAMF_MCP_AUTH_TOKEN`/`INTUNE_MCP_AUTH_TOKEN` (comma-separated to allow rotation) — the original mechanism from before Entra auth existed. **Retired in production** on 2026-07-15 once Entra OAuth was confirmed working end-to-end and no non-interactive client (n8n, scripts) was found depending on it — `IAC/ansible-servers/linux/apps/desktop-management-mcp.yml` now renders both env vars empty, so prod no longer accepts these tokens even though the code path still exists. Still used by local dev (`start-jamf.sh`/`start-intune.sh` pull the real BWS secret directly, bypassing the ansible-rendered env file) since doing an interactive Entra flow locally isn't worth the hassle for that. Grants that server's full role set (see below) when present.
2. When `ENTRA_OAUTH_ENABLED=true`, an Entra-issued JWT verified against Entra's JWKS (`src/utils/entra-jwt.ts`, using `jose`) — tool visibility is then driven by the token's `roles` claim. This is the sole auth mode in production now.

Fails closed exactly as before: if neither mode is configured for a server, every `/mcp` request gets `503`.

### Metrics

Both servers expose `GET /metrics` (Prometheus text format) via `src/utils/metrics.ts` — a fourth intentionally-shared module alongside `auth.ts`/`entra-jwt.ts`/`roles.ts`, so metric names/labels can't drift between the two server copies. It exports a `prom-client` `Registry` (with `client.collectDefaultMetrics` registered), a `metricsMiddleware(req, res, next)` Express middleware mounted via `app.use(metricsMiddleware)` right after `app.use(express.json())` (before any routes, so it also covers `/health` and `/metrics` itself), and a `metricsHandler(req, res)` route handler wired as `app.get("/metrics", metricsHandler)`. Like `/health`, `/metrics` sits outside `requireMcpAuth` (only mounted on `/mcp`) — both servers are loopback-bound in production and fronted by Caddy, so this is intentionally open the same way `/health` already is.

Metric names/labels are kept identical to the sibling `jamf-prestage-tool` repo's `server/metrics.ts` on purpose, so the same Grafana dashboards/alerts work unmodified across both apps:
- `http_requests_total` / `http_request_duration_seconds` (Counter/Histogram, labels `method`/`route`/`status`, duration buckets `[0.01, 0.05, 0.1, 0.3, 0.5, 1, 2, 5, 10]`) — inbound requests, recorded by `metricsMiddleware` off `res.on("finish")`. `route` is just `req.path` rather than a normalized route pattern: each server only ever serves `/mcp`, `/health`, `/metrics` (plus the Entra OAuth protected-resource metadata paths when `ENTRA_OAUTH_ENABLED=true`), so there's no per-request identifier (serial number, device ID, etc.) that could leak into the label and blow up cardinality.
- `external_api_request_duration_seconds` / `external_api_errors_total` (Histogram/Counter, labels `target`/`method`/`status`, duration buckets `[0.05, 0.1, 0.3, 0.5, 1, 2, 5, 10]`) — outbound calls to JAMF Pro / Microsoft Graph. Recorded inside `logApiCall()` in `src/utils/logger.ts` rather than by instrumenting `JamfClient`'s axios instance or `IntuneClient`'s Graph SDK client directly — `logApiCall(logger, method, endpoint, statusCode, duration, error?)` is the one function both `jamf-api.ts` (72 call sites) and `graph-api.ts` (63 call sites) already funnel every outbound call's method/status/duration through, so hooking it required zero changes to either API client's ~135 individual call sites. `target` (`jamf`/`intune`) is derived from the `service` string each file already passes to `createLogger('jamf-api')`/`createLogger('intune-api')`, exposed as a `logger.service` field read back inside `logApiCall`. One known gap: `jamf-api.ts`'s `catch` blocks never call `logApiCall` with an `error` argument (only `this.logger.error(...)`), so `external_api_errors_total` only increments from JAMF's rare non-2xx-but-still-logged paths, not most thrown errors — `graph-api.ts` does call `logApiCall(..., undefined, undefined, error)` on several (not all) of its error paths, which do increment the counter.
- `mcp_tool_calls_total` (Counter, labels `server`/`tool`/`status`/`caller`) / `mcp_tool_call_duration_seconds` (Histogram, labels `server`/`tool`/`status`, same duration buckets as `http_request_duration_seconds`) — per-tool-call volume, success rate, latency, and caller attribution, one level below `http_requests_total` (which only sees one `POST /mcp` per call regardless of which tool it invoked or whether the tool itself failed). Recorded by `instrumentToolCalls(server, serverLabel, caller)`, called once from each of `createJamfMcpServer`/`createIntuneMcpServer` right after constructing the `McpServer` and before any `registerTool()` calls — it monkey-patches `server.registerTool` itself so every tool registered afterward is wrapped automatically, rather than editing each of the ~113 individual `registerTool()` call sites across both files. `status` is `"error"` when the handler throws *or* when it returns normally with `isError: true` (every tool here catches its own errors internally and returns the latter shape — see `errorResult()` — so both paths have to be checked). `caller` is resolved once per request in each server's `POST /mcp` handler as `req.auth?.extra.upn ?? req.auth?.clientId ?? "unknown"` — the signed-in user's UPN for Entra-authenticated callers, else the app's `clientId`, else the literal `"static-token"` for the legacy auth path (which has no per-caller identity). `caller` is deliberately omitted from the duration histogram to keep it low-cardinality (tool × status is the useful axis for spotting a slow tool; who called it matters for volume/attribution, not latency).

### Authentication and roles

Role names are Entra App Roles defined on one shared "Desktop Management MCP" resource app registration: `Jamf.Read`, `Jamf.Write`, `Intune.Read`, `Intune.Write` (`src/utils/roles.ts`). Each server only inspects its own role names via `hasRole()`/`resolveRolesFromAuthInfo()` and ignores the rest — there's no isolation lost by sharing one resource app, since role *assignment* (who gets which role) is still granted per-role, per-user/group in Entra independently. `Intune.Write` was added 2026-07-23 alongside the first Intune write tools (Win32 app publishing/assignment landed in one change; device actions, category, Autopilot group tag, and policy/app-by-name assignment landed in a parallel one) — as of that date it exists only in `roles.ts` and needs to still be registered as an App Role in the Entra "Desktop Management MCP" app registration and assigned to testers before Entra-authenticated callers can use it (the static-token auth path already grants it automatically, same as every other role, per `resolveRolesFromAuthInfo`'s static-token-gets-everything behavior); until granted, all eight Intune write tools are unreachable via Entra auth.

Both `createJamfMcpServer(roles)` / `createIntuneMcpServer(roles)` take the caller's resolved role array and wrap each `server.registerTool(...)` call in `if (hasRole(roles, ...))` — a tool a caller's role doesn't grant is never registered on that request's `McpServer`, so it's simply not callable (not just hidden). The ten write/mutating JAMF tools (`jamf_send_mdm_command`, `jamf_update_computer`, `jamf_assign_computers_to_prestage`, `jamf_upsert_inventory_preload_record`, `jamf_flush_mdm_commands`, `jamf_create_script`, `jamf_upload_package`, `jamf_create_smart_group`, `jamf_create_policy`, `jamf_update_policy`) and the eight write/mutating Intune tools (`intune_send_device_action`, `intune_set_device_category`, `intune_set_device_name`, `intune_set_autopilot_group_tag`, `intune_assign_policy`, `intune_assign_app`, `intune_create_win32_app`, `intune_assign_app_to_groups`) additionally call `assertRole(roles, JAMF_WRITE)`/`assertRole(roles, INTUNE_WRITE)` inside their own handler as a defense-in-depth check against a future refactor accidentally omitting the registration-time gate. All other tools rely solely on the registration-time gate.

`jamf_create_script`/`jamf_upload_package`/`jamf_create_smart_group` are upserts keyed by name — re-running with the same name updates the existing object in place rather than creating a duplicate (the yearly Office/MATLAB/Adobe re-publish case). `jamf_create_policy` always creates a new policy (no upsert); `jamf_update_policy` handles the enable/disable + scope-widen/narrow workflow for an existing one. Confirmed live that Classic API PUT on policies is a **partial merge**, matching `updateComputerRecord`'s established behavior — `updatePolicyScope` reads the current policy (to merge target/exclusion group names correctly) but sends back only `general.enabled` (if changing) and `scope` in the write body, not the full object. This isn't just an optimization: echoing the *entire* current policy back (the original design) actively broke — Jamf returns several sections on GET (`printers` being the one confirmed live) that trigger a `409 Conflict` ("Problem with printer") when resubmitted verbatim on PUT. Policy/smart-group writes go to Classic API (`/JSSResource/policies`, `/JSSResource/computergroups`) as hand-built XML — confirmed live that these two endpoints reject JSON bodies with `415`, unlike `/JSSResource/computers` (used by `updateComputerRecord`), which does accept JSON. The small XML serializer lives in `jamf-api.ts` (`buildXmlDocument`/`serializeXmlObjectBody`) rather than pulling in an XML library.

**Also confirmed live:** `getSmartComputerGroups()` previously had no pagination at all, silently truncating to the API's default page (100) with no signal more existed — a tenant with >100 smart groups (this one has 125) meant any name-based lookup (`resolveComputerGroupIdByName`, `jamf_create_policy`'s group scoping, `jamf_create_smart_group`'s upsert-exists check) could silently miss a real group. Fixed to page through the full result set the same way `listManagedDevices`/`listMobileDevices` do. Separately, creating (or updating) a smart group and immediately referencing it by name in a new policy can race against Jamf's internal indexing — confirmed live as a transient `409` that resolved itself on retry a few seconds later, not a code bug. If chaining `jamf_create_smart_group` → `jamf_create_policy`/`jamf_update_policy` in one automated flow, expect to retry once on a `409` referencing a just-created group.

**Known permission gap (as of 2026-07-15):** the JAMF API client's role has Create/Update but not Delete for Scripts, Policies, or Smart Computer Groups (confirmed live — every delete attempt returned 401/403). Package deletion does work. No `jamf_delete_*` MCP tools are exposed for any of these — deletion (when needed) is a manual admin action in the JAMF Pro UI.

**Intune write tools (added 2026-07-23):** `intune_send_device_action` wraps five Graph remote actions (`syncDevice`, `rebootNow`, `remoteLock`, `retire`, `wipe`) behind one command enum, mirroring `jamf_send_mdm_command`'s shape — retire and wipe are irreversible (retire unenrolls and removes company data; wipe factory-resets). Per Microsoft's docs, `remoteLock` is Android-only and unsupported on Windows. **Known permission gap, confirmed live:** this app registration's Graph app role assignments are missing `DeviceManagementManagedDevices.PrivilegedOperations.All` — a live `syncDevice` call against a real device returned a clean 403 ("Application must have one of the following scopes: DeviceManagementManagedDevices.PrivilegedOperations.All"). That single permission gates **all five** `intune_send_device_action` commands *and* `intune_set_device_name`'s `setDeviceName` action, so every call to either tool will 403 until someone with Entra admin rights adds that Application permission to the app registration and grants admin consent — same shape as the JAMF Delete gap above, just not yet resolved. `intune_set_device_name` also calls `setDeviceName`, which is **beta-only** (no v1.0 form exists) and Windows-only.

`intune_set_device_category` assigns an existing device category to a device — confirmed live that **neither** documented approach actually works against this tenant: `deviceCategory@odata.bind` on `PATCH /managedDevices/{id}` is rejected outright ("Bind requests not supported for containment navigation property"), and Microsoft's own docs' fallback form, `PATCH .../managedDevices/{id}/deviceCategory` (also the form a still-open `msgraph-sdk-powershell` issue reports failing), 404s with an empty body. The form that actually works — verified with a live set-then-revert on a real device — is `PUT .../managedDevices/{id}/deviceCategory/$ref` with an `@odata.id` body pointing at the target category; that's what the code does. Categories themselves must already exist (no write API for creating them is exposed here, matching the "portal-only" pattern already established for JAMF buildings/departments).

`intune_set_autopilot_group_tag` resolves serial → Autopilot device identity ID the same way `getAutopilotProfileStatus` already does, then calls `updateDeviceProperties` — confirmed live with a set-then-revert on a real Autopilot-registered device. Note: the write returns 204 immediately but the change can take ~10-15s to actually show up on a subsequent read (confirmed live) — Autopilot's service has its own eventual-consistency lag independent of Graph's usual read-after-write behavior, so don't treat an immediately-stale read as a failed write.

**Two tools exist for app assignment, deliberately kept separate rather than merged into one:** `intune_assign_app_to_groups` takes the caller's full desired `{groupId, intent}[]` list and posts it verbatim — a thin, literal wrapper around Graph's actual replace-the-whole-set `/assign` semantics, correct for bulk/scripted rollouts where the caller already tracks the complete assignment set. `intune_assign_app` takes a single group (by name or GUID) plus intent and does its own read-modify-write internally to preserve every other group's existing assignment — friendlier for the common "just add/move this one group" case, at the cost of an extra read round-trip. Same tradeoff applies to `intune_assign_policy`, which has no `_to_groups`-style sibling (no bulk config-policy assignment tool exists yet).

`intune_assign_policy`/`intune_assign_app` add or move a single group's assignment on a configuration policy or app deployment. Unlike JAMF's Classic API PUT (confirmed live as a partial merge, see above), **Graph's `/assign` action fully replaces the assignment collection** rather than appending to it — confirmed live, not just read from docs. Both client methods therefore always do their own read-modify-write: GET current assignments, drop any existing entry for the same group, append the new entry, then POST the whole array back to `/assign`. **Confirmed live and fixed:** assignments can carry `source: "policySets"` when the policy/app belongs to an Intune Policy Set — echoing one of those back through `/assign` does NOT get recognized as "the same assignment already there"; it silently creates a *second*, independent `direct` assignment for that group, duplicating scope the Policy Set already grants. (Caught this live against a real Policy-Set-linked policy — the first test run left a stray duplicate direct assignment in production, which had to be manually cleaned up by re-posting the correct set.) Fixed by retaining only `source: "direct"` assignments (or ones with no `source` field at all, as classic `deviceConfigurations` assignments have) when rebuilding the array — the `policySets` entry regenerates itself automatically on the next read regardless, so it must never be included in what gets posted back. Existing retained assignments are also passed through a `sanitizeAssignmentTarget()` helper that keeps only `@odata.type`/`groupId`/`collectionId`/filter fields rather than the full GET response, as a precaution against the same class of bug JAMF's policy PUT hit (echoing an entire GET body back on write triggering a 409 from a field the API didn't expect on write) — that specific failure mode hasn't been observed on the Intune side, this is a preventive measure taken given the JAMF precedent.

**Entra ID does not support OAuth Dynamic Client Registration (RFC 7591)**, which real MCP clients lean on for zero-config setup — this project deliberately does NOT implement a custom OAuth broker/proxy to paper over that gap (see the plan history for the phased rationale); it's a resource-server-only integration. Practical per-client setup, confirmed by live testing (not just read from docs):
- **OpenCode** — works today: register a small native/public Entra client app (PKCE, no secret) and configure it as `oauth.clientId` in `opencode.json`, with an explicit `oauth.scope` (Entra's `/authorize` requires a non-empty `scope`; OpenCode doesn't infer one). `opencode mcp auth <name>` drives a normal browser PKCE flow.
- **Codex CLI** — no static-`clientId` OAuth support upstream yet; keep using `bearer_token_env_var` pointing at the existing static token. Not tied to per-user Entra identity in this phase.
- **Claude Code** — its *native* HTTP-transport OAuth (`claude mcp add --transport http ... --client-id ... --callback-port ...`) sends no `scope` parameter at all and fails against Entra with `AADSTS900144: The request body must contain the following parameter: 'scope'` — confirmed live, not an assumption; there's no CLI flag to inject one. Use the `mcp-remote` npm package instead, as a stdio↔http bridge, with **both** `--static-oauth-client-info '{"client_id":"<public-client-id>"}'` and `--static-oauth-client-metadata '{"scope":"openid profile offline_access <resource-scope>"}'` (the metadata flag is required — client-info alone doesn't carry scope). Add it via `claude mcp add-json <name> '{"type":"stdio","command":"mcp-remote","args":["<mcp-url>","<fixed-port>","--static-oauth-client-info","...","--static-oauth-client-metadata","..."]}'`. Pin an explicit port (mcp-remote otherwise derives one from a hash of the server URL) and register its exact callback redirect URI (`http://localhost:<port>/oauth/callback` — note `localhost`, not `127.0.0.1`, and note the path differs from OpenCode's own `/mcp/oauth/callback`) on the Entra public client. Revisit if/when Claude Code's native flow gains scope support upstream.

**Multiple identifier URIs are required on the resource app**, one reason this took real testing to get right: any MCP-spec client that sends the RFC 8707 `resource` parameter (OpenCode and `mcp-remote` both do, using the literal MCP server URL taken from this project's own RFC 9728 protected-resource metadata) needs that exact URL registered as an `identifierUris` entry on the Entra app — Entra rejects the request with `AADSTS9010010` if `resource` and `scope` resolve to different app identities. Since jamf-mcp and intune-mcp are different URLs sharing one resource app, both `https://jamf-mcp.colgate.edu/mcp` and `https://intune-mcp.colgate.edu/mcp` are registered as additional identifier URIs alongside the abstract `api://colgate.edu/desktop-mgmt-mcp` one.

### API Clients

- `src/jamf/jamf-api.ts` — `JamfClient`: authenticates via OAuth2 client-credentials (`POST /api/v1/oauth/token`), caches the bearer token and re-authenticates lazily via `ensureAuthenticated()` once it's within 60s of `expires_in`. Uses JAMF Pro REST API v1/v3 for most endpoints (e.g. computer lookups go through `computers-inventory` to resolve an ID, then `computers-inventory-detail/:id` for the full record); falls back to the Classic API (XML-based, e.g. policies, configuration profiles) where v1/v3 doesn't cover a resource.
- `src/intune/graph-api.ts` — `IntuneClient`: uses `@azure/identity` `ClientSecretCredential` + `@microsoft/microsoft-graph-client`'s `TokenCredentialAuthenticationProvider` (token acquisition/caching handled by the SDK, not manually). Mixes Graph API versions per-endpoint — e.g. Autopilot lookups try a `v1.0` server-side `$filter` first and fall back to fetching a `beta` batch and filtering client-side if the filter 500s. Write methods (`sendManagedDeviceAction`, `setDeviceCategory`, `setDeviceName`, `updateAutopilotGroupTag`, `assignConfigurationPolicyToGroup`, `assignAppToGroup`) live in a "Write operations" section at the bottom of the file; `resolveGroupIdByName()` (private) resolves an Azure AD group by exact `displayName` or accepts a GUID directly, used by both assignment methods.

Both clients log via `src/utils/logger.ts`, a plain `console.*`-based structured logger (`createLogger(service)` + `logApiCall`/`logAuth` helpers) — the doc comment calling it "for Azure Functions" is a holdover from where this code was extracted from; there's no Azure Functions runtime here. `LOG_LEVEL=debug` enables debug-level output.

### Device/name resolution

Read tools that take a human name generally resolve it server-side rather than requiring an ID:
- Intune: `resolveDevice()` (in `intune-server.ts`) accepts `deviceName`, `deviceId`, or `serialNumber` and normalizes to `{ deviceId, azureADDeviceId }`. `resolveAppByName()` and `resolvePolicyByName()` resolve human-readable names to GUIDs by fetching the full list and matching case-insensitively (exact match preferred, first partial match as fallback).
- JAMF: several `JamfClient` methods (e.g. `getComputerByName`, `updateComputerRecord`, `sendComputerMdmCommand`) accept a name/serial and internally look up the JAMF numeric ID before calling the detail/write endpoint.

### Tool registration pattern

Tools are registered via `server.registerTool(name, schema, handler)` with Zod schemas and `annotations` (`readOnlyHint`, `openWorldHint`, `destructiveHint`) describing side effects. Handlers return `{ content: [{ type: "text", text }] }`, or `{ isError: true, content: [...] }` on failure/not-found. Nearly every read tool takes an optional `response_format: "markdown" | "json"` (default `"markdown"`) — the shared `toText(data, format, markdownFn)` helper either dumps raw JSON or renders a hand-built markdown summary, so adding a field to markdown output means also checking whether it should surface in the raw JSON path (it already does, since JSON mode returns the full API response object, not the trimmed markdown view).

## Environment Variables

**JAMF** (injected by BWS from `bws-secrets.map`):
- `JAMF_URL` — e.g. `https://yourorg.jamfcloud.com`
- `JAMF_CLIENT_ID`, `JAMF_CLIENT_SECRET`
- `JAMF_MCP_AUTH_TOKEN` — bearer token(s) MCP clients must present (comma-separated to allow rotation). Rendered empty in prod as of 2026-07-15 (retired in favor of Entra OAuth) — still set for local dev via `start-jamf.sh`.
- `JAMF_MCP_PUBLIC_URL` — this server's externally-visible origin, e.g. `https://jamf-mcp.colgate.edu` (required only when `ENTRA_OAUTH_ENABLED=true`)
- `JAMF_PACKAGE_UPLOAD_DIR` — directory `jamf_upload_package` is allowed to read files from on this server's own filesystem (not the MCP client's). Required for that tool; it fails closed (refuses every call) if unset, and rejects any `localFilePath` outside it.

**Intune** (injected by BWS from `bws-secrets.map`):
- `AZURE_TENANT_ID`, `AZURE_CLIENT_ID`, `AZURE_CLIENT_SECRET` — Graph app-only client credentials for device data, unrelated to the Entra auth vars below
- `INTUNE_MCP_AUTH_TOKEN` — bearer token(s) MCP clients must present (comma-separated to allow rotation). Rendered empty in prod as of 2026-07-15 (retired in favor of Entra OAuth) — still set for local dev via `start-intune.sh`.
- `INTUNE_MCP_PUBLIC_URL` — this server's externally-visible origin, e.g. `https://intune-mcp.colgate.edu` (required only when `ENTRA_OAUTH_ENABLED=true`)

**Entra ID auth** (shared by both servers, each can enable independently; injected by BWS from `bws-secrets.map`):
- `ENTRA_OAUTH_ENABLED` — `"true"` to accept Entra-issued bearer tokens on `/mcp` in addition to the static token
- `ENTRA_TENANT_ID` — Entra tenant GUID (deliberately separate from `AZURE_TENANT_ID` above — different app registration, different purpose)
- `ENTRA_RESOURCE_APP_ID_URI` — Application ID URI of the "Desktop Management MCP" resource app, e.g. `api://colgate.edu/desktop-mgmt-mcp` (Entra tenants that require a verified domain in the URI, as Colgate's does, will reject a bare `api://desktop-mgmt-mcp` — confirmed live)
- `ENTRA_RESOURCE_APP_ID` — GUID app ID of the same resource app; accepted as an alternate `aud` value. MCP-spec clients that send the RFC 8707 `resource` parameter (e.g. OpenCode) get tokens audienced to this GUID rather than the URI above, even with `requestedAccessTokenVersion: 2` set on the app — confirmed against a real issued token, not assumed from docs.

## Tests

`test/jamf-api.test.ts` uses Node.js built-in `node:test` — no extra test framework. Tests are live integration tests against a real JAMF Pro API. Write operations are gated behind `JAMF_TEST_WRITE=1`. Tests that may fail due to API client permissions use `permissionAwareTest()` which treats 401/403/404 as a skip rather than a failure.

The script/smart-group/policy upsert write tests deliberately do NOT create-then-delete (the API client can't delete those object types — see the permission gap noted above); each looks for a pre-existing, manually-created fixture by name (`TEST_SCRIPT_NAME`, `TEST_SMART_GROUP_NAME`, `TEST_POLICY_NAME` — all optional, test skips via `t.skip()` if unset or if no object with that name exists yet) and only exercises the update-in-place path. The package upsert test is fully self-cleaning instead (`TEST_PACKAGE_PATH`, also optional) since package delete permission is confirmed present.

`test/auth.test.ts` covers `requireMcpAuth`'s dual-mode fallback/fail-closed behavior and the `roles.ts` helpers with fakes — no live credentials or network access needed, hence the separate `npm run test:unit` script.

## Deployment

Both servers run in production as Podman quadlet containers on `podman02` (built from the repo-root `Dockerfile`, one image, two containers — the Intune container overrides the image's default `CMD` via the quadlet's `Exec=`), fronted by Caddy which terminates TLS and reverse-proxies to the container's plain-HTTP port. The servers themselves never see TLS directly. Client-facing URLs are `https://jamf-mcp.colgate.edu/mcp` and `https://intune-mcp.colgate.edu/mcp` — internal DNS only, no public exposure, and several JAMF tools are destructive, so `/mcp` also requires a bearer token (see Environment Variables) on top of the network restriction — defense in depth, not a substitute for it. Deploy/redeploy playbooks live in `IAC/ansible-servers/linux/apps/desktop-management-mcp.yml` (full deploy) and `desktop-management-mcp-update.yml` (code-only redeploy); see `IAC/CLAUDE.md` for the broader Terraform → Ansible dispatch pipeline.

Both quadlets are rendered from **one shared env file** (`/etc/desktop-management-mcp/desktop-management-mcp.env`, one Ansible template, one set of vars) — so despite `ENTRA_OAUTH_ENABLED` reading as a per-server toggle in code, there is currently no way to enable Entra auth for jamf-mcp without also enabling it for intune-mcp, or vice versa. This is a known gap versus the original rollout plan (which assumed independent enablement), left as-is because it's low-risk while role assignment is still solo-testing scale — no one has an `Intune.*` role yet. Splitting into two env files (or two env-var names) would be needed to truly decouple this before a wider pilot.
